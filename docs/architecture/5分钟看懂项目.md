# 5 分钟看懂项目

## 这个项目要解决什么问题

我们有很多设备，型号不同，但本质接近：

- 底层是 Linux
- 上层带 AI 模块
- 很多设备带 JT808
- 都需要调试 USB、网络、串口、配置、日志、平台参数、白名单、视频能力

所以这个项目不是为某一台设备做工具，而是做一个“可扩展的设备调试平台”。

## 为什么不按设备型号开发

如果按型号开发：

- 新来一款设备就重写一遍
- 代码会越来越散
- 后期视频、JT808、AI 参数会互相缠绕

所以这里采用：

- Connector：负责怎么连设备
- DeviceProfile：负责连上后怎么理解设备
- Feature：负责用户最终要完成什么任务

## 这个项目最重要的三个抽象

### 1. Connector
负责接入方式：

- ADB
- RNDIS
- 串口
- SSH
- HTTP
- Raw USB

### 2. DeviceProfile
负责设备画像：

- 配置文件在哪里
- 日志在哪里
- 哪些字段是终端 ID / 车牌 / 平台地址 / 白名单
- 用什么规则检查设备状态

### 3. Feature
负责最终用户功能：

- 设备发现
- 终端
- 配置中心
- 日志中心
- 诊断中心
- JT808 中心
- 视频中心

## 首期为什么先做 ADB / RNDIS / 串口

因为这三条链路最常见、最实用：

- ADB：最强，最适合 Linux 设备调试
- RNDIS：很多设备插上就会出网卡，方便扫 IP 和探服务
- 串口：系统起不来、ADB 没开时还能救场

## 首期成功标准

第一版不求全，但至少要做到：

- 能识别设备
- 能判断它支持哪种进入方式
- 能用一种方式进入设备
- 能看终端输出
- 能读到关键配置
- 能导出基础诊断信息

## 后期怎么扩展

后期新增设备时，优先补：

- 新 Connector
- 新 DeviceProfile
- 新字段映射

而不是推翻整个工程。
