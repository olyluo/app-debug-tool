# 字段分层策略说明

## 一、目标

避免把平台字段抽象误解成“所有设备必须完全遵守的硬标准”。

这份文档用于说明：

- 哪些字段只是平台参考抽象
- 哪些字段来自设备画像映射
- 哪些字段属于厂商扩展
- 哪些字段属于运行态推导结果

## 二、推荐分层

### 1. Reference Fields（参考字段）

这层是平台内部统一展示、统一处理时使用的参考字段。

特点：

- 命名稳定
- 语义明确
- 方便 UI、诊断、导出使用
- 不要求所有设备必须完整具备

示例：

- `terminalId`
- `licensePlateNo`
- `mainServerAddr1`
- `avWhitelist`

### 2. Profile Mapped Fields（画像映射字段）

这层是某个具体设备画像已经确认可以映射出来的字段。

特点：

- 与某类设备强相关
- 来自 profile 规则
- 可能并不完整
- 不同设备映射结果可能不同

示例：

- 某设备把 `licenseplateno` 映射到 `licensePlateNo`
- 某设备把 `mainserveaddr1` 映射到 `mainServerAddr1`

### 3. Vendor Extended Fields（厂商扩展字段）

这层用于容纳：

- 无法稳定归入参考字段的内容
- 厂商私有语义字段
- 短期内还不适合进入统一抽象的字段

特点：

- 允许命名保留厂商特征
- 可随着理解加深逐步提升为参考字段

示例：

- 某厂商私有 AI 参数
- 某厂家专用视频控制项
- 某类私有业务状态字段

### 4. Runtime Derived Fields（运行态推导字段）

这层表示不是直接读配置，而是通过运行态探测、网络扫描、服务检查得到的字段。

特点：

- 可能随设备状态变化
- 更接近当前真实能力
- 常用于能力判断和诊断

示例：

- `wifiHotspotEnabled`
- `rndisIp`
- `videoPreviewSupported`
- `adbAvailable`

## 三、推荐数据流

```text
设备原始字段 / 数据源
 -> Profile 解析与映射
 -> 参考字段 + 厂商扩展字段
 -> 再叠加运行态推导字段
 -> 最终供 UI / 诊断 / 导出使用
```

## 四、设计原则

- 参考字段服务于统一展示，不是强制标准
- 画像映射优先于硬编码规则
- 扩展字段允许长期存在
- 运行态字段和配置字段要分开管理

## 五、团队使用建议

当新增一款设备时，优先问这几个问题：

1. 它能映射到哪些参考字段？
2. 哪些字段只能暂时放在厂商扩展层？
3. 哪些能力只能靠运行态探测？
4. 哪些字段未来值得提升为参考字段？
